## What This Step Does

Visualization is the final step. It takes all outputs from Steps 2–5 — structural metrics, RMAV prediction scores, impact scores, validation metrics — and synthesizes them into interactive dashboards. The goal is to move from numbers to decisions.

```
Steps 2–5 Outputs                Visualization               Output
──────────────────               ─────────────               ──────
M(v)  — metric vectors           Pipeline         →  Static HTML dashboard
Q(v)  — prediction scores           │                (archivable evaluation artifact)
I(v)  — impact scores               │             →  Genieus live web app
Spearman ρ, F1, …             ──────┘                (operational practitioner tool)
```

The dashboard design follows one principle: every visual element should reduce time-to-decision for a specific architectural concern. Each view answers a different stakeholder question; the [From Dashboard to Decisions](#from-dashboard-to-decisions) section maps questions to views.

---

## Two Visualization Surfaces

| Surface | Format | Best For |
|---------|--------|---------|
| **Static HTML Dashboard** | Self-contained `.html` file | Research documentation, CI artifacts, offline review, sharing with stakeholders |
| **Genieus Web App** | Next.js + FastAPI live app | Real-time interactive exploration, triggering pipeline steps, collaborative team review |

Both surfaces display the same seven information sections. The static dashboard is generated by `bin/visualize_graph.py` and embeds all data inline. The live app connects to Neo4j in real time and allows triggering analysis, simulation, and validation directly from the browser.

## Static HTML Dashboard

### Section 1 — Executive Overview

KPI cards summarizing the analysis at a glance:

| KPI | Definition |
|-----|-----------|
| **Total components** | Count of analysed components in the selected layer(s) |
| **CRITICAL** | Components with Q(v) ≥ upper fence (box-plot) |
| **SPOFs detected** | Components with AP_c > 0 (structural single points of failure) |
| **Anti-patterns** | Total architectural problems detected by the ProblemDetector |
| **Validation status** | PASS ✓ / FAIL ✗ based on primary gates from Step 5 |
| **Spearman ρ** | Overall correlation between Q(v) and I(v) |

These six numbers give a system health summary in under ten seconds. A system with 0 SPOFs, ≤ 3 CRITICAL components, 0 CRITICAL anti-patterns, and PASS validation is architecturally sound from a criticality standpoint.

### Section 2 — Layer Comparison

Side-by-side bar charts comparing RMAV dimension distributions (Reliability, Maintainability, Availability, Vulnerability) across all analysed layers. Allows architects to identify which layer has the most pronounced structural concerns.

The layer comparison answers: "Is our availability problem mostly an infrastructure issue, or an application-level coupling issue?" If the `infra` layer shows much higher A(v) averages than `app`, the bottleneck is physical colocation, not logical dependency.

### Section 3 — Component Details Table

Sortable and filterable table of all components with full RMAV breakdown:

| Column | Description |
|--------|-------------|
| **Rank** | Sorted by Q(v) descending |
| **Component** | ID and type |
| **Level** | CRITICAL / HIGH / MEDIUM / LOW / MINIMAL badge |
| **Q(v)** | Composite prediction score |
| **R(v)** | Reliability dimension |
| **M(v)** | Maintainability dimension |
| **A(v)** | Availability dimension |
| **V(v)** | Vulnerability dimension |
| **I(v)** | Simulated impact score (if Step 4 was run) |
| **SPOF** | ✓ if AP_c > 0 |
| **Patterns** | Anti-pattern labels |

Clicking a component row in the live app opens the full detail panel with cascade sequence, dependency list, and remediation recommendations.

### Section 4 — Correlation Scatter Plot

Q(v) vs I(v) scatter plot — the **visual proof** of the methodology's central claim.

Each point represents a component. The horizontal axis is the predicted Q(v) score; the vertical axis is the simulated impact I(v). A tight cluster around the diagonal means the predictions match reality. Points in the upper-left quadrant (high I, low Q) are **false negatives** — underestimated critical components. Points in the lower-right (high Q, low I) are **false positives**.

A tight cluster around the diagonal with ρ ≥ 0.70 is the visual target.

### Section 5 — Interactive Network Graph

A force-directed graph rendered with vis.js. The layout organizes components such that highly connected components (high centrality) migrate toward the centre — the layout itself becomes a qualitative criticality indicator.

| Action | Effect |
|--------|--------|
| Hover | Tooltip showing component type, Q(v), I(v), criticality level, SPOF flag |
| Click | Highlights all direct DEPENDS_ON neighbours; dims the rest |
| Double-click | Centres and zooms to the selected node |
| Drag | Repositions the node (layout adapts) |
| Scroll | Zoom in/out |

Use the network graph for systems up to ~80 components. Above that threshold, switch to the Dependency Matrix (Section 6), which scales without visual saturation.

### Section 6 — Dependency Matrix

A directed adjacency matrix $A$, where $A_{ij} = w(e)$ if a dependency exists from component $i$ to component $j$. Components are ordered using the **Reverse Cuthill-McKee (RCM)** algorithm to bring non-zero cells as close to the diagonal as possible. In this topological ordering, tightly coupled clusters emerge as dense diagonal blocks.

**Reading the matrix:**
- **Blocks along the diagonal:** Tightly coupled clusters — assess as a unit for redundancy planning.
- **Full rows:** High out-degree component — many efferent couplings (M(v) risk).
- **Full columns:** High in-degree component — many others depend on it (R(v) risk).

### Section 7 — Validation Report

Displays all eleven validation metrics from Step 5 with pass/fail badges:

- **Primary gates** (Spearman ρ, p-value, F1-Score, Top-5 Overlap): shown with ✓/✗ indicators
- **Secondary gate** (RMSE): shown with ✓/✗
- **Reported metrics** (Kendall τ, Pearson r, Precision, Recall, Cohen's κ, Top-10 Overlap, NDCG@K, MAE): shown without gates

The validation report answers: "Can I trust the Q(v) predictions in this dashboard?" If all primary gates pass, use the predictions confidently. If any gate fails, review the [Interpreting Results](validation.md#interpreting-results) section before acting on the predictions.

## Genieus: Live Web Application

A Next.js 15 application that communicates with the FastAPI backend (port 8000) and Neo4j database (port 7687). All pipeline operations can be triggered from the browser without running CLI commands.

**Five-tab structure:**

| Tab | Primary View | Function |
|-----|-------------|----------|
| Dashboard | KPI cards, criticality pie, top-10 list, validation badges | System health at a glance |
| Graph Explorer | Interactive 2D/3D force-directed dependency graph | Explore topology, filter by layer, inspect components |
| Analysis | Layer selector, weight mode toggle, real-time results | Trigger and view Steps 2 & 3 |
| Simulation | Component selector, failure mode picker, cascade animation | Trigger and view Step 4 |
| Settings | Neo4j URI, credentials, database name | Connection configuration |

### Graph Explorer Tab

The Graph Explorer is the primary architectural review interface. It provides:

- **Layer filter**: app / infra / mw / system
- **Search**: highlight and centre on a specific component
- **Criticality overlay**: nodes colored and sized by CRITICAL / HIGH / MEDIUM / LOW / MINIMAL
- **Type overlay**: nodes shaped by Application / Broker / Node / Topic / Library
- **Component panel** (on click): full RMAV scores, I(v), criticality level, cascade count, direct dependency list
- **2D / 3D toggle**: three-dimensional layout for dense graphs (> 100 components)

### Analysis Tab

Triggers Steps 2 and 3 directly from the browser:
- Select layer: `app`, `infra`, `mw`, `system`
- Toggle weight mode: `AHP-derived` vs `equal`
- View results table with full RMAV breakdown and anti-pattern annotations
- Export results as JSON

### Simulation Tab

Triggers Step 4:
- Select target component from dropdown
- Choose failure mode: CRASH / DEGRADED / PARTITION / OVERLOAD
- Animated cascade overlay shows which components fail in sequence
- Impact metrics displayed in real time

## Visualization Design Rationale

**Why force-directed layout?** The force model positions high-betweenness components near the centre of the graph automatically. Without any manual positioning, structurally important components cluster together at the visual centre — making criticality patterns immediately apparent before any colour coding is applied.

**Why RCM for the dependency matrix?** Random or alphabetical ordering scatters dependency structure arbitrarily, hiding real coupling patterns. RCM minimizes matrix bandwidth, bringing tightly coupled clusters onto the diagonal. This reveals genuine architectural structure at a glance.

**Why colour by criticality level rather than raw score?** Showing five discrete levels (CRITICAL through MINIMAL) instead of a continuous gradient makes relative risk immediately legible. An architect scanning the dashboard can instantly identify all CRITICAL components without needing to reason about score magnitudes.

**Why embed all data in the HTML?** The static dashboard is a research artifact — it must be archivable, shareable with stakeholders who have no infrastructure, and reproducible months after the analysis was run. A self-contained HTML file with embedded data satisfies all three requirements without any dependencies.

## Anti-Pattern Detection

Anti-pattern findings from the ProblemDetector (Step 3 sub-step) are surfaced in the dashboard in three ways:

- **KPI card**: "Anti-patterns detected: N" in the Executive Overview
- **Component table**: anti-pattern labels (SPOF, GOD, BOTTLENECK, SYSTEMIC) in the Patterns column
- **Network graph**: SPOF components receive a dashed border; GOD components receive an oversized node

Clicking an anti-pattern label in the component table opens a detail panel explaining the topological signature, the operational risk it represents, and the recommended remediation strategy.

## From Dashboard to Decisions

| Question | Which View |
|----------|-----------|
| "What are the most critical components I must protect?" | Executive Overview + Component Table (rank-sorted) |
| "Which components are structural SPOFs?" | Component Table (filter: SPOF = ✓) + Network Graph |
| "Is our reliability concern in the app layer or infra layer?" | Layer Comparison — R(v) bars |
| "Do our predictions actually correlate with failure impact?" | Validation Report + Scatter Plot |
| "What happens if DataRouter fails?" | Simulation Tab → select DataRouter → CRASH |
| "Which components depend on this broker?" | Graph Explorer → click broker → highlight dependents |
| "Are there hidden cyclic dependencies?" | Dependency Matrix (off-diagonal symmetric blocks) |
| "Is the architecture getting worse over time?" | Compare dashboards from successive CI runs |

## Performance

| System Scale | Dashboard Generation Time |
|---|---|
| tiny (8 components) | < 1s |
| small (15 components) | ~2s |
| medium (35 components) | ~5s |
| large (80 components) | ~12s |
| xlarge (200 components) | ~40s |

For xlarge systems, set `include_network=False` to skip the vis.js network graph (the dominant rendering cost). The dependency matrix, scatter plot, and component table still render quickly.

## Commands

```bash
# Generate dashboard from Neo4j data
python bin/visualize_graph.py --layers app system --output output/dashboard.html

# Open immediately in browser
python bin/visualize_graph.py --layers app --open

# Skip network graph for large systems (faster)
python bin/visualize_graph.py --layers system --no-network --output output/dashboard.html

# Full pipeline — generate and visualize in one pass
python bin/run.py --all --layer app --open
```

## Programmatic API

```python
from src.core import create_repository
from src.analysis import AnalysisService
from src.simulation import SimulationService
from src.validation import ValidationService
from src.visualization import VisualizationService

repo       = create_repository()  # uses env vars for Neo4j connection
analysis   = AnalysisService(repo)
simulation = SimulationService(repo)
validation = ValidationService(analysis, simulation)

viz = VisualizationService(
    analysis_service=analysis,
    simulation_service=simulation,
    validation_service=validation,
    repository=repo
)

output_path = viz.generate_dashboard(
    output_file="output/dashboard.html",
    layers=["app", "system"],
    include_network=True,    # set False for large systems
    include_matrix=True,
    include_validation=True,
)

print(f"Dashboard saved to: {output_path}")
repo.close()
```

See `examples/example_visualization.py` for a complete runnable example.

## Methodology Summary

Step 6 closes the methodology loop by making all six steps' outputs accessible to the humans who act on them. Together the pipeline moves from raw architecture description to informed, quantified decisions:

| Step | Output | Answers |
|------|--------|---------|
| 1. Modeling | G(V, E, w) | What does the system look like as a graph? |
| 2. Analysis | M(v) — 13 metrics per component | Where is each component positioned topologically? |
| 3. Prediction | Q(v), RMAV, criticality level | Which components are structurally at-risk, and why? |
| 4. Simulation | I(v) — empirical impact per component | Which components actually cause the most damage when they fail? |
| 5. Validation | Spearman ρ, F1, pass/fail | Can we trust the topology-based predictions? |
| **6. Visualization** | **Dashboard** | **Where should we act, and what should we do?** |
